## Phase 142 — Memory Subsystem Connections (Gap A/B/C)

### Background

Seven memory subsystems existed in the codebase but did not talk to each other:
1. vectorStore (CTC sidecar)
2. graphTraversal / graph_claim_edges (MySQL)
3. contradictionDetector / contradictionAlerts (MySQL, weekly scan)
4. trainingBridge + trainingCorpusListener (CLF pipeline)
5. verificationEventStore (in-memory ring buffer)
6. coordApi / memoryRouter (KG memory, MySQL coordContext)
7. MRAgent episodic memory (evolva-mragent HTTP server) ← new in Phase 141

Three gaps were identified and closed:

### Gap A — MRAgent contradictions → contradictionAlerts DB

**File:** `server/mrAgentContradictionPersist.ts`

Real-time MRAgent contradiction detections are now persisted to the `contradictionAlerts` DB table using the same schema as the weekly `contradictionDetector.ts` scan. Severity is auto-derived from similarity score (≥ 0.95 → high, ≥ 0.85 → medium, else low). Existing open alerts are updated (upsert); resolved/dismissed alerts are skipped.

Wired into `analysisPipeline.ts` post-verdict hook (non-blocking, fires after `checkMrAgentContradiction`).

### Gap B — trainingExporter → trainingBridge (Channel 3)

**File:** `server/trainingExporter.ts` (updated)

`exportHighConfidenceVerdict` now calls `emitVerdictEvent()` via `trainingBridge` as a third channel alongside MRAgent ingest and CLF JSONL. High-confidence verdicts flow through the established CLF LoRA training pipeline automatically — no duplicate write path.

`emitVerdictEvent` signature: `{ claimText, verdict, rationale, sourceUrl?, domain? }` — `rationale` carries `"Confidence: 0.XXX"`.

### Gap C — claimSimilarityEngine → MRAgent episodic pass

**File:** `server/claimSimilarityEngine.ts` (updated)

`findSimilarClaims()` now accepts `includeMrAgentEpisodic?: boolean` in `SimilarityOptions`. When enabled (defaults to `ENV.mrAgentEnabled`), queries MRAgent `/query` for recently verified similar claims and merges them with TF-IDF DB results. Runs even when DB corpus is empty. Deduplicates by `claimId`. Re-sorts by similarity after merge.

Two helper functions extracted to keep `findSimilarClaims` complexity ≤ 20 (ESLint gate):
- `parseEpisodeToClaim(ep)` — parses episode text format `"VERDICT: X\nCLAIM: Y"`
- `mergeEpisodicResults(dbResults, queryText, topK)` — async merge + dedup + sort
